# 📋 Plan: POS Dashboard UI/UX Redesign with shadcn/vue

## Context
The POS dashboard is a critical point-of-sale interface for a vape store. It features a two-column layout (products left, cart right) with mobile drawer support. The current implementation uses custom CSS variables, inline styles, and hand-rolled components. This plan outlines a systematic redesign using shadcn/vue components and UI/UX Pro Max principles to improve consistency, accessibility, and maintainability.

---

## Assumptions

### ✅ Confirmed
- Project uses Laravel + Inertia.js + Vue 3 + TypeScript
- Tailwind v4 with CSS variables enabled (`--tw` theme system)
- shadcn/vue (reka-ui based) is installed with New York v4 style, neutral base
- Icon library: Lucide
- Existing custom POS CSS variables in `app.css` under `.pos-layout`
- Dashboard is used in a fast-paced retail environment (cashier workflow)
- Primary user: Indonesian (IDR currency, "Rp" format)
- Color scheme: Teal (#14b8a6) primary, slate/navy secondary
- Responsive breakpoints: mobile (<1024px hides right panel), tablet, desktop

### ⚠️ Clarifications Needed
1. **Component state management**: Should modals (Discount, Payment, Receipt) also be migrated to shadcn Dialog/Sheet? Or kept separate?
2. **Animation requirements**: Current has animations (fade-in, zoom-in). Should we keep custom animations or use shadcn's built-in?
3. **Empty state branding**: Current uses generic SVG (`/images/icon/no-task.svg`). Should we replace with shadcn Empty/illustration component?
4. **Cart button in header**: The cart toggle (shopping-cart-21/22.svg swap) — should this become a lucide icon with badge?
5. **Design system migration strategy**: Should we migrate in phases (search first, then categories, then products, then cart) or all at once?
6. **Performance**: Product grid can have hundreds of items. Should we implement virtual scrolling (`@vueuse/core`'s `useVirtualList`)?
7. **Accessibility compliance target**: WCAG 2.1 AA? Keyboard navigation for all interactive elements?
8. **Dark mode**: The POS has forced light mode via `html.dark .pos-layout`. Should we support dark POS theme or leave as-is?
9. **Form validation**: Payment modal uses `v-model.number` without validation. Should we add shadcn Form + zod validation?
10. **Existing shadcn usage**: None found in current POS components. Should we adopt `Button.vue`, `Badge.vue`, `Card.vue`, `Input.vue`, `Sheet.vue` (drawer), `Dialog.vue` (modals) universally?

## Task Breakdown

### Phase 0: 100vh Layout Foundation Fix (CRITICAL — MUST BE FIRST)

**Problem**: Cart panel terpotong, product grid scroll tidak sampai ujung karena `calc(100vh - Npx)` yang brittle dan konflik dengan struktur flex.

**Root cause**: `<slot />` di `PosLayout.vue` tidak memiliki `flex-1 min-h-0`, sehingga dashboard page tidak memenuhi sisa tinggi layar. Ini membuat perhitungan `100vh` menjadi tidak valid.

**Solution**: Pure flexbox — no `calc(100vh - Npx)` anywhere.

---

#### Task 0.1: PosLayout.vue — wrapper slot dengan flex-1

**File**: `resources/js/layouts/pos/PosLayout.vue` (line 5-12)

**Change**:
```vue
<!-- BEFORE -->
<SidebarInset>
  <PosHeader v-if="cashier" ... />
  <slot />
</SidebarInset>

<!-- AFTER -->
<SidebarInset>
  <PosHeader v-if="cashier" ... />
  <div class="flex-1 min-h-0 overflow-hidden">
    <slot />
  </div>
</SidebarInset>
```

**Why**: `SidebarInset` adalah `flex-1 flex-col` (shadcn default). Untuk `<slot />` mengisi sisa ruang setelah header, dibutuhkan wrapper dengan `flex-1 min-h-0`.

## PHASE 0: 100vh Layout Foundation Fix (CRITICAL)

**⚠️ This phase MUST be completed BEFORE any other phase.** The current layout uses brittle `calc(100vh - XXXpx)` formulas that break on different screen sizes and content changes. This phase implements a robust flexbox-based 100vh layout that guarantees both panels fill the viewport exactly.

### Objective
- Root container: `height: 100vh` (full viewport)
- Left panel (products): scrolls internally, fills available height
- Right panel (cart): stretches to bottom without overflow, no internal scroll
- Mobile: cart drawer fills height via Sheet component

### Root Cause Analysis

**Dashboard structure (current):**
```
PosLayout (min-h-screen)
  └─ SidebarInset (flex col)
      ├─ PosHeader (fixed height ≈ 60px)
      └─ <main class="flex flex-1 min-h-0 overflow-hidden">  ← GOOD
          ├─ Left: flex-1 flex-col  ← GOOD
          │   └─ Product grid: maxHeight: calc(100vh - 220px)  ← BRITTLE
          └─ Right: aside with height: calc(100vh - 712px)   ← BRITTLE
```

The `712px` = header height + search/category heights + cart header + summary + button + all padding/margins. Any change breaks it.

### Solution: Pure Flexbox

**Key principle**: Let flexbox distribute space naturally. No magic numbers.

**Layout changes:**

1. **PosLayout.vue** — already `min-h-screen`, but ensure full height constraint:
   - Root `.pos-layout` should have `min-h-screen` (already) but we need `h-screen` on the flex container that wraps `<main>`.
   - In `PosLayout.vue`: `<SidebarInset>` should be `flex-1 flex flex-col min-h-0` (already likely). Verify.

2. **Dashboard.vue** — `<main>` container stays as `flex flex-1 min-h-0 overflow-hidden`. GOOD.

3. **Left products column** (inside main):
   - Currently: `<div class="flex-1 min-h-0 flex flex-col bg-... overflow-hidden">` — GOOD.
   - The **search+categories section** is fixed height (auto, based on content). FLEX START.
   - **Product grid**: `flex-1 overflow-y-auto` — NO `max-height` or `min-height` inline styles.
     - Remove: `:style="{ maxHeight: 'calc(100vh - 220px)', minHeight: 'calc(100vh - 220px)' }"`
     - Use: `class="flex-1 overflow-y-auto"` (or `flex-1 min-h-0 overflow-y-auto` for safe flex child)
   - This makes the grid fill remaining vertical space after header (search+categories) and scroll internally.

4. **Right panel (desktop cart)**:
   - Current: `:style="{ height: 'calc(100vh - 712px)' }"` — REMOVE.
   - Replace with: `class="h-full flex-shrink-0"` (or `h-full` alone if parent has fixed height).
   - BUT: parent `<main>` has `flex-1 min-h-0 overflow-hidden` and is itself a flex child of `SidebarInset`. To make `h-full` work, the `<main>` must have a defined height. Since `<main>` is `flex-1`, it fills remaining space. The right panel as `h-full` will match `<main>`'s height.
   - ALSO: Right panel needs to be a flex **sibling** to left, so both are direct children of `<main>` (already true).
   - Remove from `CartPanel.vue` the fixed `maxHeight: 400px` on items container — use `flex-1 min-h-0 overflow-y-auto` to allow natural fill.

5. **CartPanel.vue** — internal height management:
   - Root Card: already `h-full flex flex-col` (we'll enforce in Task 4.1).
   - Cart items container: `class="flex-1 min-h-0 overflow-y-auto py-1 px-4 space-y-2"` — NO fixed maxHeight.
   - Summary + Total + Button: `shrink-0` (stay at bottom).
   - This ensures cart panel takes full height, items scroll if needed, total button always visible.

6. **CartDrawer.vue** — Sheet (mobile):
   - shadcn `SheetContent` with `side="right"` and `class="h-full w-full max-w-[320px]"` ensures it fills viewport height.
   - No extra height calculations needed.

7. **PosLayout.vue** — ensure the flex chain:
   ```vue
   <div class="pos-layout h-screen flex flex-col min-h-screen">
     <SidebarProvider>
       <PosSidebar />
       <SidebarInset class="flex-1 flex flex-col min-h-0">
         <PosHeader v-if="cashier" />
         <slot class="flex-1 min-h-0" />  <!-- dashboard page gets this -->
       </SidebarInset>
     </SidebarProvider>
   </div>
   ```
   - The `<slot>` (dashboard) needs `flex-1 min-h-0` to fill remaining space below header.
   - The `<main>` inside dashboard already `flex-1 min-h-0`, so it fills that slot.
   - Both panels inside `<main>` are flex children that can use `h-full` or `flex-1`.
